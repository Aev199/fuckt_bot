# Geotech Knowledge Bot + Web Cabinet

## Product Format

Продукт состоит из двух интерфейсов:

- основной интерфейс: Telegram-бот в формате pseudo-SPA
- дополнительный интерфейс: web-кабинет для просмотра, навигации и ограниченного редактирования

## Product Goal

Система должна помогать:

- быстро сохранять полезную информацию по геотехнике
- потом удобно находить её по структуре и поиску
- просматривать материалы как справочник
- редактировать материалы только доверенным пользователям

## UX Principle

### Telegram bot is primary for capture

Бот остаётся основным интерфейсом для быстрых действий:

- добавить материал
- открыть категории
- найти материал
- открыть карточку материала
- прикрепить фото

### Web cabinet is primary for browsing

Web должен стать не формой редактирования, а удобным справочником.

Основной сценарий web:

- выбрать категорию или подкатегорию
- увидеть список материалов
- открыть материал и спокойно читать
- при наличии прав перейти в режим редактирования

## Access Model

### Read access

Просмотр через web доступен:

- владельцу через Telegram auth
- администратору через `WEB_CABINET_TOKEN`

### Edit access

Редактировать через web могут только пользователи, чьи `telegram_id` входят в белый список.

Для этого в конфиг добавляется переменная:

- `WEB_EDITOR_TELEGRAM_IDS`

Формат:

```env
WEB_EDITOR_TELEGRAM_IDS=371122920,123456789
```

Если пользователь не входит в этот список:

- он может просматривать материалы
- не может изменять, удалять и создавать материалы или категории через web

## Category Model

В структуре категорий разрешён только один уровень вложенности:

- категория
- подкатегория

Глубже вложенность запрещена.

Примеры:

- `Грунты -> Суглинки`
- `Фундаменты -> Ленточные`
- `Испытания -> Статическое зондирование`

## Main Web Layout

Web-кабинет должен стать трёхпанельным.

### Left column: category tree

Содержит:

- список категорий
- раскрывающиеся подкатегории
- счётчик материалов
- кнопки фильтрации по выбранной ветке

### Center column: materials list

Содержит:

- строку поиска
- быстрые фильтры
- список материалов

Быстрые фильтры:

- `Все`
- `Избранное`
- `С фото`
- `Последние`

Каждая карточка в списке должна показывать:

- заголовок
- короткий фрагмент
- категорию / подкатегорию
- теги
- источник
- количество фото
- дату обновления

### Right column: material detail

Содержит:

- заголовок
- категорию и подкатегорию
- источник
- основной текст
- заметки
- теги
- фотографии
- действия

По умолчанию правая колонка работает как режим просмотра.

## Edit Mode

Редактирование не должно быть главным состоянием страницы.

Логика:

1. пользователь открывает материал
2. видит его в режиме чтения
3. если у него есть права, он видит кнопку `Редактировать`
4. по нажатию detail-pane переключается в edit mode

В режиме редактирования доступны:

- заголовок
- текст
- источник
- заметки
- категория
- подкатегория
- теги
- удаление материала

Если прав нет:

- кнопки редактирования скрыты
- интерфейс остаётся read-only

## Material View Requirements

Карточка материала в web должна быть удобной для чтения.

Обязательные блоки:

- `Заголовок`
- `Категория / Подкатегория`
- `Источник`
- `Содержимое`
- `Заметки`
- `Теги`
- `Фото`

Если каких-то данных нет, блок либо скрывается, либо показывается компактно.

## Attachments

Поддерживаются фото-вложения.

В web должно быть:

- сетка превью
- открытие полного изображения
- удаление фото только при наличии edit rights

## Search and Filtering

Поиск должен работать по:

- заголовку
- содержимому
- заметкам
- источнику
- тегам

Фильтры должны поддерживать:

- категорию
- подкатегорию
- избранное
- наличие фото
- последние материалы

## API Changes Required

### Category API

Нужно добавить или расширить:

- получение дерева категорий
- создание подкатегории с `parent_id`
- валидацию глубины вложенности

### Material API

Нужно добавить:

- фильтр `has_attachments`
- фильтр по `parent_category_id` / `subcategory_id`
- явный флаг прав редактирования в ответе `/api/auth/me` или отдельном endpoint

## Frontend State Model

Минимальное состояние frontend:

- текущий пользователь
- есть ли права редактирования
- дерево категорий
- выбранная категория
- выбранная подкатегория
- текущий фильтр списка
- поисковая строка
- выбранный материал
- режим detail pane: `view | edit`

## Visual Direction

Web должен ощущаться как рабочий кабинет, а не как техно-форма.

Ориентиры:

- спокойный светлый интерфейс
- плотная, но читаемая структура
- акцент на чтении и поиске
- чёткая визуальная иерархия
- минимум модальных окон

## Out of Scope for This Iteration

В этот этап не входят:

- drag-and-drop сортировка категорий
- произвольная глубина дерева
- публичные ссылки на материалы
- полноценная ролевая система
- редактор rich text

## Definition of Done

Итерация считается завершённой, когда:

1. web-кабинет корректно сохраняет изменения
2. web-кабинет удобен для чтения материалов
3. есть дерево `категория -> подкатегория`
4. редактирование доступно только `WEB_EDITOR_TELEGRAM_IDS`
5. обычный пользователь без прав может безопасно пользоваться web как справочником
